2026-03-17 architecture, dashboard, storage

Overview

This project is being organized as a continuous research system rather than a pile of notes. The goal is to make it easy to add glaze recipes, images, firing details, process observations, and provenance while also making it possible for a new thread or agent to synthesize what already exists.

Key Decisions

  • Markdown with TOML front matter is the canonical store for recipes, sources, actions, and reports.
  • SQLite is generated from those records so the project can query by cone, surface quality, source, and status.
  • Cone remains central in the schema by participating in the recipe primary key.
  • Every report written in markdown gets a generated HTML companion.
  • Every research website visit is tracked in an append-only log plus normalized source records.

Operating Model

Each research pass should:

  1. Log the source.
  2. Capture recipe and image records.
  3. Write or update a report that explains what was learned.
  4. Rebuild the database and dashboard.

Next Phase

The next practical step is to ingest a first wave of authoritative and high-signal glaze sources, then start populating cone 6 and cone 10 recipes with image-backed records.
